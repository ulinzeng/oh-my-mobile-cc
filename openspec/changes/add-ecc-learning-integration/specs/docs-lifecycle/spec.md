# docs-lifecycle (modified)

## ADDED Requirements

### Requirement: ECC-observed skill integration

The project MUST be able to ingest ECC `continuous-learning-v2` evolved
skills into the project's `docs/skills/drafts/` tree with schema
normalization, provenance tracking, and secret redaction.

Evolved skills generated by ECC MUST land in `docs/skills/drafts/` (not
the root `docs/skills/` tree). Promotion to `docs/skills/` is a manual
`mv` + human review step — never automatic.

Each imported skill MUST carry:
- `name`, `description`, `triggers`, `related-specs` — the existing
  project schema.
- `evolved-from: <absolute-or-relative-path-to-ecc-source>`
- `evolved-confidence: <float 0..1>` when present in the source.
- `draft: true` until a human promotes the skill.

Before writing, absolute paths (`/Users/...`, `/home/...`), obvious API
keys (e.g. `AKIA[0-9A-Z]{16}`), and `.env` content fragments MUST be
redacted.

#### Scenario: Minimal ECC skill imports cleanly
- **GIVEN** an ECC evolved skill at
  `~/.claude/homunculus/projects/1b870545265a/evolved/skills/x.md`
  with `id: x` and `trigger: "foo"`
- **WHEN** `.claude/scripts/ecc-skill-to-docs.sh` runs on it
- **THEN** `docs/skills/drafts/x.md` is created with
  `name: x` and `triggers: [foo]` and `draft: true`
- **AND** an `evolved-from` provenance line points back to the source

#### Scenario: Pre-set related-specs is preserved, not overwritten
- **GIVEN** an ECC evolved skill whose front-matter already contains
  `related-specs: [specs/pairing/spec.md]`
- **WHEN** the converter runs
- **THEN** the output keeps `related-specs: [specs/pairing/spec.md]`

#### Scenario: Secrets are redacted
- **GIVEN** an ECC evolved skill whose body contains
  `/Users/alice/.env` and `AKIAIOSFODNN7EXAMPLE`
- **WHEN** the converter runs
- **THEN** the output body contains neither the path nor the key

### Requirement: Automated cluster-to-badcase mapping

The project MUST automatically write a bad-case file to
`docs/badcases/<YYYY-MM-DD>-auto-<hash>.md` when the failure-cluster
check detects an unresolved TDD stall (≥ 3 `[red]` commits with 0
`[green]` commits). The generated bad-case MUST carry
`status: auto-generated`, MUST carry `signal-source: [red-cluster]` (or
an equivalent signal tag), MUST list the offending commit subjects in
the body, and MUST be idempotent — re-running on the same cluster hash
neither overwrites nor duplicates the file.

#### Scenario: Three-red cluster produces one badcase
- **GIVEN** a session window with commits whose subjects include three
  `[red]` tags and zero `[green]` tags
- **WHEN** `.claude/scripts/session-failure-cluster-check.sh` runs
- **THEN** a new file `docs/badcases/<today>-auto-<hash7>.md` is created
- **AND** its front-matter contains `status: auto-generated`

#### Scenario: Healthy red-then-green cycle writes nothing
- **GIVEN** a session window with one `[red]` and one `[green]` commit
- **WHEN** `.claude/scripts/session-failure-cluster-check.sh` runs
- **THEN** no file is created under `docs/badcases/`

#### Scenario: Re-running on an already-auto-generated cluster is idempotent
- **GIVEN** a badcase `docs/badcases/<date>-auto-<hash>.md` already exists
- **WHEN** the same cluster signal fires again in a later session
- **THEN** the existing file is not rewritten
- **AND** no duplicate file is created
