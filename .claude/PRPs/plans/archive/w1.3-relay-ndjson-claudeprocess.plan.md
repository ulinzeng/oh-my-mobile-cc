# Plan W1.3: ClaudeProcess + NDJSON Flow (relay edge)

> Sub-plan of `.claude/PRPs/plans/kmp-claude-code-remote.plan.md`, Phase W1, Task 1.3.
> Prerequisite: W1.1 (✅ merged — `CCEvent`, `WireMessage`, `ProtocolJson`,
> `NdjsonFramer` shipped on `feat/w1-protocol`, coverage ≥ 97.8%).
> Scope is tight: only the `relay` module. No mobile UI, no Ktor WebSocket
> (that's W1.5), no ApprovalBridge (that's W1.4, gated by a new OpenSpec
> change proposal).

## Summary

Ship the JVM-only `relay` process wrapper that:

1. Spawns `claude -p --output-format stream-json --input-format stream-json`
   with a clean environment and a pinned working directory.
2. Exposes **three** structured streams from the child process:
   - `events: Flow<CCEvent>` — decoded NDJSON from stdout.
   - `stderr: Flow<String>` — raw stderr lines, kept out-of-band (never
     mixed into the event stream; logged at warn level).
   - `exit: Deferred<Int>` — terminal exit code, completes exactly once.
3. Accepts a `writeUserMessage(json: JsonObject)` entry point that
   serializes a single `{"type":"user",...}` stream-json frame to CC stdin
   followed by `\n` + `flush`. Anything else (e.g. permission responses)
   is **refused** at compile time via a sealed input type — per the
   protocol spec, permission decisions never flow through stdin.

Together with `Flow<String>.ccEvents()` from W1.1, this closes the loop:
raw child-process stdout chunks → NDJSON lines → typed `CCEvent`s ready
for `ApprovalBridge` (W1.4) and the mobile relay (W1.5) to consume.

This is the last relay-internal piece before we can write an end-to-end
approval flow integration test in W1.4.

## Patterns to Mirror

### MIRROR-1: Process I/O with structured concurrency

Standard JVM `ProcessBuilder` + `java.lang.Process` wrapped in coroutines.
Reference shape (do **not** copy verbatim; adapt for this project):

```kotlin
// relay/src/main/kotlin/io/ohmymobilecc/relay/claude/ClaudeProcess.kt
public class ClaudeProcess(
    private val command: List<String>,
    private val workingDir: Path,
    private val env: Map<String, String>,
    private val scope: CoroutineScope,
    private val json: Json = ProtocolJson.default,
) : AutoCloseable {
    private val process: Process = ProcessBuilder(command)
        .directory(workingDir.toFile())
        .redirectErrorStream(false) // GOTCHA-1: MUST be false
        .also { it.environment().apply { clear(); putAll(env) } }
        .start()

    public val events: Flow<CCEvent> = process.inputStream.asLineFlow()
        .ccEvents(json)
        .flowOn(Dispatchers.IO)

    public val stderr: Flow<String> = process.errorStream.asLineFlow()
        .flowOn(Dispatchers.IO)

    public val exit: Deferred<Int> = scope.async(Dispatchers.IO) {
        process.waitFor()
    }

    public fun writeUserMessage(message: ClaudeInput.UserMessage) {
        val line = json.encodeToString(JsonObject.serializer(), message.toJsonObject()) + "\n"
        // GOTCHA-2: one write + flush per line, holding a mutex so the
        // writer isn't interleaved.
        writeMutex.withLock {
            process.outputStream.write(line.toByteArray(Charsets.UTF_8))
            process.outputStream.flush()
        }
    }

    override fun close() {
        runCatching { process.outputStream.close() }
        if (process.isAlive) process.destroy()
    }
}
```

**GOTCHA-1**: `redirectErrorStream` MUST be `false`. If stderr is merged
into stdout the `ccEvents()` parser will throw on CC log lines.
Spec: `openspec/specs/protocol/spec.md` Requirement "CC 事件解码" —
unknown types are Unknown, but bare stderr text is **not JSON** at all.

**GOTCHA-2**: `process.outputStream` is a `BufferedOutputStream`. Without
`flush()` CC will block waiting for its stdin to reach a newline boundary
on the kernel side. Always write `\n` and then `flush()`.

**GOTCHA-3**: Coroutine cancellation does **not** kill the child. On
`close()` we must also call `process.destroy()`. Do NOT rely on
`scope.cancel()` alone.

**GOTCHA-4**: `process.waitFor()` is blocking — wrap it in
`scope.async(Dispatchers.IO)` and expose as `Deferred<Int>`. Callers that
want a non-blocking poll use `exit.isCompleted`.

**GOTCHA-5**: On JVM, `InputStream.bufferedReader().lineSequence()` is
synchronous. Convert to a cold `Flow<String>` with a dedicated
`Dispatchers.IO` consumer so cancellation of the downstream Flow
terminates the reader thread without leaking it.

### MIRROR-2: Sealed input type prevents permission regressions

Per the updated protocol spec (post-ADR-0004, post-archive of
`fix-approval-bridge-mechanism`), permission decisions MUST flow through
the `PreToolUse` hook exit, never through CC stdin. Encode this at the
type level so no future refactor can silently regress:

```kotlin
// relay/src/main/kotlin/io/ohmymobilecc/relay/claude/ClaudeInput.kt
public sealed class ClaudeInput {
    public abstract fun toJsonObject(): JsonObject

    public data class UserMessage(
        val role: String = "user",
        val content: String,
    ) : ClaudeInput() {
        override fun toJsonObject(): JsonObject = buildJsonObject {
            put("type", JsonPrimitive("user"))
            put("message", buildJsonObject {
                put("role", JsonPrimitive(role))
                put("content", JsonPrimitive(content))
            })
        }
    }
    // Intentionally NO PermissionResponse variant — protocol forbids it.
}
```

Then `writeUserMessage` takes `ClaudeInput` (public API) but the only
concrete subclass is `UserMessage`; a future contributor trying to add
`PermissionResponse` must also change the spec (ADR-0004 still forbids),
which is the forcing function we want.

### Edge cases checklist

- [ ] Child prints a stderr log line that **looks like** JSON (e.g. `{"warn":"..."}`) → lands in `stderr` Flow, **not** in `events`.
- [ ] Child exits immediately after spawn → `events` Flow completes, `exit.await() == 0` (or the child's real exit code), no hang.
- [ ] Child exits mid-line (partial JSON trailing) → partial dropped (matches `ndjsonLines()` contract).
- [ ] UTF-8 multi-byte character split across read chunks → joined correctly by `InputStreamReader`.
- [ ] `writeUserMessage` called concurrently from two coroutines → mutex serializes; no byte interleaving.

## Complexity / Risk

- **Complexity**: M — process I/O + coroutines + fixture-driven testing
  is standard but has sharp edges (GOTCHA-1 through -5).
- **Confidence**: High — pattern is well-worn; the fixture lets us test
  deterministically without spawning real `claude`.
- **Key risk**: platform differences — macOS vs Linux signal numbers,
  `bash` location (`/bin/bash` vs `/usr/bin/env bash`). Mitigation: use
  `ProcessBuilder.command("bash", "-c", "...")` and tolerate non-zero
  exit on `close()` without asserting a specific code.
