# Plan W1.4: ApprovalBridge + `relay-cli approval-bridge` subcommand

> Sub-plan of `.claude/PRPs/plans/kmp-claude-code-remote.plan.md`, Phase W1, Task 1.4.
> Prerequisites: W1.1 (✅ merged) and W1.3 (✅ merged — `ClaudeProcess`,
> `ClaudeInput` on main). OpenSpec already codifies the mechanism in
> `openspec/specs/approval-inbox/spec.md` + `protocol/spec.md` (post
> `fix-approval-bridge-mechanism` archive, see ADR-0004) — **no new
> change proposal needed**, this is pure implementation.
>
> Scope is tight: only the `relay` module. No Ktor WebSocket (W1.5), no
> SqlDelight persistence (W2.3 owns that — we use an in-memory store
> behind a port, swappable later), no mobile UI (W2).

## Summary

Ship the two halves of the approval bridge:

1. **Main relay side (`ApprovalBridge`)** — in-process orchestrator:
   - Accepts `PreToolUse` hook requests coming in over local IPC
   - Allocates `approvalId`, persists to a `ApprovalStore` port
     (InMemory for now; W2.3 swaps in SqlDelight)
   - Emits `WireMessage.ApprovalRequested` on an outbound
     `SharedFlow<WireMessage>` (W1.5 will bridge this to WS)
   - Blocks per-request until either a `WireMessage.ApprovalResponded`
     arrives (from inbound channel) or 10 min elapse → auto-DENY
   - Returns the resolved `Decision` back to the waiting IPC peer
   - Broadcasts `ApprovalExpired` on timeout
2. **Child subcommand side (`relay-cli approval-bridge`)** —
   stateless RPC client:
   - Parses argv (`--session-id`)
   - Reads the CC hook payload JSON from stdin
   - Dials local IPC → forwards payload → blocks on decision
   - Prints the CC-format decision JSON to stdout + `\n`, exits 0
   - On IPC failure: stderr diagnostic, exit != 0 (CC treats as deny)

Together with W1.3's `ClaudeProcess`, this means a fully local relay
can already drive real `claude -p` with `PreToolUse` hook approvals
gated on a programmatic `Decision` source (the unit tests just feed
decisions directly; W1.5 will feed them over WS).

## Patterns to Mirror

### MIRROR-1: Ports & Adapters

The store and the WireMessage emission are ports. Concrete adapters:
`InMemoryApprovalStore` (for W1.4 + tests) and later `SqlDelightApprovalStore`
(W2.3). This keeps the SQLite dependency out of W1.4 entirely.

```kotlin
// relay/src/main/kotlin/io/ohmymobilecc/relay/approval/ApprovalStore.kt
public interface ApprovalStore {
    public suspend fun insertPending(row: ApprovalRow)
    public suspend fun updateState(approvalId: String, state: ApprovalState, decisionJson: String?)
    public suspend fun findById(approvalId: String): ApprovalRow?
    public suspend fun policyExists(tool: String, sessionId: String): Boolean
}
```

### MIRROR-2: IPC choice — UNIX domain socket (not TCP)

**Decision (in this plan, with rationale):** use a UNIX domain socket at
`$XDG_RUNTIME_DIR/oh-my-mobile-cc/relay.sock` (fallback
`$TMPDIR/ohmymobilecc-relay-$UID.sock`).

Why UDS over `127.0.0.1`:
- Permissions via filesystem mode `0600` scope it to the current UID;
  a TCP port on localhost can be reached by any local process.
- No port-conflict surface; one less config knob.
- JDK 17+ has first-class `java.net.UnixDomainSocketAddress`, so no JNI.
- Windows: JDK 17+ supports UDS on Win10 1809+, so developer parity
  holds.

**GOTCHA**: JDK 17's `AFUNIXSocketChannel.open(StandardProtocolFamily.UNIX)`
requires NIO; we wrap it in blocking `InputStream`/`OutputStream` via
`Channels.newInputStream` for simplicity — the subprocess side is a
short-lived one-shot RPC, no throughput concern.

**GOTCHA**: Stale socket files after a crash block re-bind. Delete the
socket file on startup (after a lock-file check that we're the only
relay) so a crash-restart cycle is clean.

### MIRROR-3: Wire format for the local RPC

We do NOT reuse `WireMessage` here — that's the mobile↔relay protocol.
The bridge RPC is relay-internal and tiny: NDJSON with two ops,
`request` (child→main) and `decision` (main→child). Same framing
(`\n` separator), no versioning yet (internal API; rev with a bump when
we add auth / multi-tenancy in later phases).

```kotlin
// relay/src/main/kotlin/io/ohmymobilecc/relay/approval/BridgeRpc.kt
@Serializable
internal data class BridgeRequest(
    val sessionId: String,
    val toolName: String,
    val toolUseId: String,
    val toolInput: JsonObject,
    val hookPayload: JsonObject, // raw hook JSON for forensics
)

@Serializable
internal data class BridgeDecision(
    val permissionDecision: String,          // "allow" | "deny"
    val permissionDecisionReason: String,
    val updatedInput: JsonObject? = null,    // set when CUSTOMIZE
)
```

### MIRROR-4: Virtual-clock for testing the 10-minute timeout

We will NOT `Thread.sleep` 10 min in tests. Inject a `TimeSource` abstraction
(or, more cheaply, take an instant provider `() -> Instant` and a
`delay: suspend (Long) -> Unit` seam) so `ApprovalBridgeTest` can drive
time with `kotlinx.coroutines.test.TestCoroutineScheduler.advanceTimeBy()`.
The production wiring passes `kotlinx.datetime.Clock.System` and the real
`kotlinx.coroutines.delay`.

### MIRROR-5: Test structure

Consistent with W1.3: kotlin.test + JUnit Platform + `runTest` where we
control time, `runBlocking` where we spawn real subprocesses. Fixture
driven — use a hook-payload JSON from
`shared/src/commonTest/resources/fixtures/real_captures/03-hook-bridge-approved.ndjson`
(extract the `hook_started` PreToolUse line).

### Edge cases checklist

- [ ] Two concurrent `requestApproval`s with same `toolUseId` — must
      not collide on `approvalId` (UUID each)
- [ ] `submitDecision` for unknown `approvalId` — no-op + log warn
- [ ] `ApprovalBridge.close()` cancels all in-flight waiters with
      `ApprovalExpired(reason="shutdown")` (this is new — spec allows
      `reason` extension; test that it doesn't throw)
- [ ] UDS path longer than OS limit (~104 bytes on macOS, ~108 on
      Linux) — fail fast with a clear error rather than mid-flight
- [ ] Hook payload missing `tool_use_id` — request rejected with
      non-zero exit from the subcommand

## Complexity / Risk

- **Complexity**: M-H — UDS + virtual-clock timeouts + subprocess
  end-to-end test stack up. Every piece is well-trodden but the
  composition needs discipline.
- **Confidence**: High — spec is crisp (ADR-0004 spent W0 closeout
  stress-testing the idea against real `claude`), fixture 03 is live
  data, IPC surface is small.
- **Key risks**:
  - **UDS on Windows CI**: deferred — we're macOS/Linux only for W1.
    Add `@DisabledOnOs(OS.WINDOWS)` on `BridgeServerClientTest` and
    log a TODO for W1.5+.
  - **Virtual time precision in TestCoroutineScheduler**: using
    `delay(600_000)` inside the bridge with a custom `Delay` seam
    avoids the classic "runTest ignores unclocked withTimeout" trap
    (hit in W1.3).
  - **Test flakiness on `ApprovalBridgeCommandTest`**: spawning a
    JVM from a JVM is slow and env-sensitive. Mitigation: run the
    command in-process via a programmatic entry point first; keep the
    full subprocess test as a single optional smoke case.

## What this plan is NOT doing

- No Ktor WebSocket, no auth, no pairing — W1.5.
- No SqlDelight — W2.3. We use an in-memory store behind an interface;
  the swap is a single class.
- No mobile UI. The `SharedFlow<WireMessage>` output is observable but
  has no consumer yet.
- No `settings.local.json` injection by relay — that's a packaging
  concern for W4 polish. For W1.4, tests wire the bridge directly;
  real `claude` integration is left to manual smoke when W1.5 lands
  the WS.
- No multi-session isolation beyond what `sessionId` already provides;
  socket is single-process, single-user.
